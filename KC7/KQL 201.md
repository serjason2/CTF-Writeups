# Section 1: Time-Based Filtering
**In KQL 201, I'll learn**

- **Aggregation** - Go beyond the simple counts with `dcount()` to find unique values, `min()` and `max()` to track timelines, and `make_set()` to group related indicators

Along the way, I'll investigate a password spray attack and use these techniques to uncover the attacker's pattern.

In cybersecurity investigations, when something happened is just as important as what happened.

**When investigating, I'll often need to find events that occur before or after a specific time. KQL uses standard comparison operators with timestamps:**
| Operator | Meaning                         | Example                                      |
|----------|---------------------------------|----------------------------------------------|
| `>`      | After a specific time           | Find logins after March 10                   |
| `<`      | Before a specific time          | Find failed logins before March 10           |
| `>=`     | On or after a specific time     | Find events from March 10 onward             |
| `<=`     | On or before a specific time    | Find events up to and including March 10     |


**Example: Finds all processes executed after a specific timestamp**
```
ProcessEvents
| where timestamp > datetime(2024-06-17T14:49:02Z)
```
- This is useful when you're identifying a suspicious event and want to see what happened next.


When working with timestamps, logs often store events down to millisecond. The `startofday()` function rounds timestamps down to midnight, making it easier to analyze a full day's activity
```
AuthenticationEvents
| where timestamp >= startofday(datetime(2024-05-01))
| where timestamp < startofday(datetime(2024-05-02))
```
- This filters all authentication events on May 1, 2024 from 12:00 AM to 11:59 PM


The `endofday()` function rounds timestamps up to 11:59:59.999 PM. Combined with `<=`, I can include the entire final day in a range.
```
AuthenticationEvents
| where timestamp <= endofday(datetime(2024-05-07))
| where result == "Failed Login"
```
- This finds all failed logins on or before May 7, 2024

The `between` operator is a cleaner way to filter for a date range:
```
AuthenticationEvents
| where timestamp between (datetime(2024-05-01) .. datetime(2024-05-07))
| where result == “Failed Login”
```
- The `..` defines the range. Both endpoints are inclusive. This is often easier to read than using multiple comparison operators
- So far we’ve used specific dates like `datetime(2024-05-01)`. But in a real SOC, you’d want to look at “the last 24 hours” without hardcoding dates.


The `ago()` function calculates a timestamp relative to the current time:
```
AuthenticationEvents
| where timestamp > ago(24h)
```
- **Common time units**: `m` (minutes), `h` (hours), `d` (days)
- **Note**: In KC7 training dataset, dates are fixed to a specific time period. But in production environments, `ago()` is your go-to for time-based filtering


The `bin()` function groups timestamps into buckets, making it easy to spot spikes in activity:
```
AuthenticationEvents
| where result == “Failed Login”
| summarize hourly_failures = count() by bin(timestamp, 1h)
| order by timestamp asc
```
- This group failed logins into 1-hour buckets. A sudden spike from 5 failures per hour to 500 is a red flag worth investigating,.

**Common intervals**: `1m` (1 minute), `5m` (5 minutes), `1h` (1 hour), `1d` (1 day).

# Section 2: Aggregation
- With so many results returned, some attacks blend in with normal activity. That’s why aggregation is key to spotting unusual patterns.
- A single failed login isn’t suspicious. But 100 failed logins from one IP? That’s worth investigating.
- The `summarize` operator transforms raw events into meaningful insights by grouping and counting data

Let’s count how many successful logins came from each IP address
```
AuthenticationEvents
| where result == “Successful Login”
| summarize total_logins = count() by src_ip
```
- This groups all successful logins by source IP and counts them. Each row in the result shows one IP and its login count.

I can group by multiple fields for more detailed analysis:
```
AuthenticationEvents
| where result == “Failed Login”
| summarize failed_attempts = count() by username, src_ip
```
- This shows failed attempts grouped by both username AND IP address together. I can see if one IP is attacking multiple accounts, or if multiple IPs are all targeting one account.

Sometimes counting total events isn’t enough. I need to know how many unique values exist.

- The `dcount()` function (distinct count) can help me answer questions like:
    - “How many unique accounts did a suspicious IP try to access?”
    - “How many different hosts ran a particular process?”

**Example:**
```
AuthenticationEvents
| where result == “Failed Login”
| summarize unique_accounts = dcount(username) by src_ip
| where unique_accounts > 1
```
- This query finds IPs that targeted multiple accounts. I will notice that 2 IPs are trying to login via 20 different usernames. That’s possibly a password spray attack

Checking to see if same passwords were being reused
```
AuthenticationEvents
| where timestamp between (datetime(2024-05-01) .. datetime(2024-05-07))
| where result == “Failed Login”
| summarize count() by password_hash
```
- This counts how many times each password was used in failed login attempts (password spray)
- That’s a lot of passwords. Hard to see any patterns with 2000+ results

Let’s sort them to find which passwords were used most often:
```
AuthenticationEvents
| where timestamp between (datetime(2024-05-01) .. datetime(2024-05-07))
| where result == “Failed Login”
| summarize count() by password_hash
| sort by count_ desc
```
- `desc` keyword means descending (highest first)
- `asc` keyword means ascending (lowest first)
- Some passwords are being reused a lot. The key question is are they being used against the same account over and over, or sprayed across many different accounts?

Let’s use `dcount()` to find out:
```
AuthenticationEvents
| where timestamp between (datetime(2024-05-01) .. datetime(2024-05-07))
| where result == “Failed Login”
| summarize unique_accounts = dcount(username) by password_hash
| sort by unique_accounts desc
| where unique_accounts > 10
```
- If the same password is being used against many different accounts, that’s a classic password spray attack
- No password spray detected, but it’s good that I confirmed it.


**Notice how I built my analysis step by step:**
1. First I counted passwords used in failed logins
2. Then I sorted to find the most-used ones
3. Then I switched to `dcount()` to see how many accounts each password targeted

This iterative approach is key to using KQL investigations, it applies to most other query languages as well. I can also add comments to document my thinking by using `//` 

I can calculate multiple aggregations in a single query:
```
AuthenticationEvents
| where result == “Failed Login”
| summarize
total_attempts = count(),
unique_passwords = dcount(password_hash),
first_seen = min(timestamp),
last_seen = max(timestamp)
by src_ip
```
- This shows total attempts, unique passwords tried, and the time range of activity for each IP.
- The `min()` function finds the earliest timestamp. The `max()` function finds the latest.

When did an attack start? When did it end? The min() and max() functions help track timelines:
```
Email
| where link has “baddomain.com”
| summarize
first_seen = min(timestamp),
last_seen = max(timestamp),
by sender
```
- This shows when each sender first and last sent emails containing that link. If the timestamps are months apart, that attack has been ongoing for a long time

The `make_set()` function collects all unique values into a list, instead of just counting them:
```
Email
| where link has “baddomain.com”
| summarize recipients = make_set(recipient) by subject
```
- Instead of just counting recipients, this shows the actual list of who was targeted by each email subject. If one subject was sent to dozens of people, that’s a mass phishing campaign.

Let's apply this to browsing activity. A bunch of users we seen going to the malicious domain `nothing-to-see-here.net`. One of them downloaded a suspicious tool from that domain.
```
OutboundNetworkEvents
| where url has  "n_____________"
| summarize _______(url) by _____
```

The `top` operator combines sorting and limiting into one step:
```
ProcessEvents
| summarize execution_count = count() by process_name
| top 10 by execution_count
```
- This returns the 10 most executed processes

# Section 3: Data Transformation
Now let’s learn how to shape my query output

When I run a query, I get every column in the table. But often only I care about a few fields. The `project` operator selects which column to display:
```
AuthenticationEvents
| take 100
| where result == “Failed Login”
| project timestamp, username, src_ip
```
- This returns only the three columns you need, making output cleaner and easier to read.

I can also rename columns with `project`:
```
AuthenticationEvents
| take 100
| project EventTime = timestamp, User = username, SourceIP = src_ip
```
- This creates more readable column names. Useful when sharing results with teammates or creating reports.


The `extend` operator adds new columns without removing existing ones:
```
AuthenticationEvents
| take 100
| extend hour_of_day = hourofday(timestamp)
```
- This adds a column showing the hour (0-23) while keeping all other columns intact.

**Key difference:**

- `project` - replaces columns with only the ones you specify
- `extend` - keeps everything and adds new columns


The `iff()` function adds conditional logic:
```
AuthenticationEvents
| extend status = iff(result == “Successful Login”, “OK”, “FAILED”)
```
- Syntax: `iff(condition, value_if_true, value_if_false)` 


For multiple conditions, use `case()` :
```
| extend risk = case(
attempts > 100, "CRITICAL",
attempts > 50, "HIGH",
attempts > 10, "MEDIUM",
"LOW"
)
```
- The last value is the default when no conditions match
